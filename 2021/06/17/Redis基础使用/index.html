4. 持久化策略

  1. RDB（默认，内存充裕推荐使用，快照为dump.rdb）

优点：备份，还原速度快，高效，相当于照快照，适合灾难备份

缺点：启动RDB策略有一定规则，可能随时启动，会占用较大内存

启动条件：

​ 1.正常关闭，./redis/bin/redis-cli shutdown

​ 2.配置文件的：

​ save 900 1 15分钟至少有1个key发生改变

​ save 300 10 5分钟至少有10个key发生改变

​ save 60 10000 1分钟至少有10000个key发生改变

  1. AOF（通过日志记录改变key的语句来备份，备份文件appendonly.aof）

优点：持续占用很小的内存来运行备份程序

缺点：日志文件比较大，不适合灾难备份

备份策略：每秒备份（默认，不安全 ）、每修改备份（建议使用）、不备份

改appendonly为yes

6. 哨兵模式（sentinel）

  • 一主二从三哨兵（建立在主从模式的基础上，因此也需要用slaveof先设定好主从的关系），新建sentinel.conf
  • 启动哨兵
  • 优点：解决了主从复制的宕机后，只能读取的问题，且重启后，不需要手动确定主从关系（所哨兵都宕机除外）
  • 主服务器宕机后，主节点的选取机制：

    • 不需要所有哨兵一起选举，只需要选举出一个Leader哨兵，由Leader哨兵再选举出主节点即可，选举Leader哨兵的方法（raft算法）：

      • 按照“先下手为强”的本质，每个哨兵每一轮都只有1票，每个哨兵分别为自己拉票，通常是最先发现主节点宕机的哨兵先发起拉票，票数超一半则选举结束，成为Leader哨兵，否则进入第二轮选举，所以通常哨兵个数为奇数。

    • Leader哨兵选举主节点的规则（优先级 > offset > run id）：

      • 优先级最高的，配置文件中slave-priority的值较小的，默认情况下都相同；
      • 偏移量offset最大的，代表从主节点同步的进度，越大代表数据量越接近主节点；
      • run id最小的，每个redis实例启动后都会随机生成一个40位的run id；
